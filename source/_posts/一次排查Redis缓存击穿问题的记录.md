缓存问题在后端开发里非常常见，但真正自己去定位、去分析、去修复一次之后，理解才会深很多。最近在复盘高并发场景时，我把缓存击穿这个问题重新梳理了一遍，也顺手总结成这篇文章。

<!-- more -->

## 这篇讲什么

这篇文章主要记录一个典型问题：热点数据失效后，大量请求同时打到数据库，导致数据库压力瞬间升高。文章会从问题现象、排查过程、解决方案三个角度来整理。

## 我遇到的问题

在一个商品查询场景里，系统本来是先查 Redis，没有命中再查数据库并回填缓存。正常情况下，这种流程没有问题，但如果某个热点商品缓存刚好过期，而这时又有大量请求同时进来，就会出现下面这种情况：

- 多个请求同时发现缓存未命中
- 它们几乎同时去访问数据库
- 数据库短时间承受大量重复查询
- 响应时间明显上升，严重时会影响整个系统稳定性

这个问题本质上就是缓存击穿。

## 我是怎么排查的

一开始看到的是“数据库 QPS 突然升高”，但应用层报错并不明显，所以排查时不能只盯着代码，而要把请求链路完整串起来。

我当时的排查思路大致是这样的：

1. 先看 Redis 命中率是否下降。  
如果命中率在某个时间点突然变低，说明大量请求绕过了缓存。

2. 再看数据库慢查询和连接数。  
如果数据库查询量明显上升，而且 SQL 语句高度重复，就基本能判断是缓存层出了问题。

3. 对照热点 key 的过期时间。  
如果异常时间点刚好和热点 key 失效重合，那么问题就更加明确了。

排查到这里，其实就能确认：不是数据库本身先出问题，而是缓存失效后的并发访问把数据库打高了。

## 最终解决方案

针对缓存击穿，常见的解决思路是“让同一时刻只有少量请求真正去查数据库”，我这里整理了三种比较实用的办法。

### 1. 互斥锁

当缓存失效时，只允许一个线程去查数据库并重建缓存，其他线程等待或快速失败。

```java
String cache = redisTemplate.opsForValue().get(key);
if (cache != null) {
    return cache;
}

String lockKey = "lock:" + key;
boolean locked = tryLock(lockKey);
if (!locked) {
    Thread.sleep(50);
    return queryWithCache(key);
}

try {
    String again = redisTemplate.opsForValue().get(key);
    if (again != null) {
        return again;
    }
    String dbResult = queryFromDb(key);
    redisTemplate.opsForValue().set(key, dbResult, 30, TimeUnit.MINUTES);
    return dbResult;
} finally {
    unlock(lockKey);
}
```

这种方式的核心思想是“重建缓存这件事要收口”，避免所有请求一起打数据库。

### 2. 热点数据不过期

如果某些数据访问量非常高，而且变化频率不高，可以考虑让逻辑过期而不是物理过期。也就是说，缓存本身一直保留，只是在后台异步刷新。

它的优点是用户请求不会因为缓存重建而直接打到数据库，代价是实现会复杂一些，需要自己维护数据的新旧状态。

### 3. 给过期时间加随机值

如果很多缓存 key 的过期时间完全一致，那么它们可能在同一时间大面积失效，造成流量集中穿透。给 TTL 增加一点随机值，可以把失效时间打散。

```java
int baseTtl = 1800;
int randomTtl = ThreadLocalRandom.current().nextInt(300);
redisTemplate.opsForValue().set(key, value, baseTtl + randomTtl, TimeUnit.SECONDS);
```

这个办法不能单独解决所有问题，但作为低成本优化很有效。

## 我学到了什么

这次整理让我更清楚地意识到，缓存不是“加了就一定快”，而是“加对了才稳定”。很多缓存问题不是功能错误，而是系统在高并发场景下的行为变化。

缓存击穿背后最核心的点有两个：

- 热点数据是少数，但影响很大
- 问题往往发生在缓存失效那一瞬间

所以设计缓存方案时，不能只看“平时能不能命中”，还要看“失效时系统会发生什么”。

## 总结

如果一句话总结缓存击穿，我会这样理解：

缓存击穿不是 Redis 没有发挥作用，而是在最关键的时刻，太多请求一起绕过了它。

在后端开发里，这种问题很典型，也很值得反复复盘。因为你每多理解一层缓存行为，对高并发系统的把握就会更扎实一些。
