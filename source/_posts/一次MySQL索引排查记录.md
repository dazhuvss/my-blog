很多时候数据库变慢，并不是业务逻辑突然复杂了，而是某条原本还能接受的 SQL，在数据量上来之后开始暴露问题。最近我在复盘数据库相关知识时，把一次典型的索引排查过程重新整理了一遍。

<!-- more -->

## 这篇讲什么

这篇文章想记录一个很常见但也很值得反复复盘的问题：一条查询语句为什么会突然变慢，以及我是怎么一步步把问题定位到索引上的。

## 我遇到的问题

场景并不复杂：一个商品列表查询接口，在测试数据不多的时候响应一直正常，但当数据量增加之后，接口耗时开始明显上升。应用侧没有报错，数据库也没有直接报警，所以问题最开始看起来并不明显。

后来把 SQL 单独拿出来分析，发现它虽然“能查出来”，但执行效率并不稳定。

## 我是怎么排查的

我排查这类问题一般会按这个顺序来：

1. 先看慢查询日志，确认是不是某条 SQL 明显拖慢了整体响应。
2. 对 SQL 执行 `EXPLAIN`，先看有没有走索引。
3. 再看过滤条件、排序字段、联合索引顺序是否匹配。
4. 最后再判断是不是回表、范围查询或索引失效造成的问题。

很多时候，问题并不在“有没有索引”，而在“索引是否真的被当前 SQL 用到了”。

## 最终定位

这次的核心问题是：虽然表上有索引，但 SQL 的条件组合和排序方式没有很好命中联合索引，导致数据库最终选择了代价更高的执行计划。

具体看，常见的几个坑包括：

- 联合索引没有满足最左前缀原则
- 条件里对字段做了函数处理，导致索引失效
- `order by` 和 `where` 的组合没有和索引顺序对齐
- 查询字段过多，导致回表成本高

## 我学到了什么

复盘下来，我更清楚地意识到：索引不是“建了就完事”，而是要和查询方式一起设计。

判断一条 SQL 性能是否稳定，不能只看它“能不能跑”，而要看：

- 执行计划是否稳定
- 过滤条件是否真正利用了索引
- 排序、分页是否会放大开销

## 总结

以后再看慢 SQL，我会先把思路收敛到这几步：看日志、跑 `EXPLAIN`、核对索引、再反推 SQL 写法。这样比一上来盲改语句高效得多。
