# Agent项目

## 推理模式（ReAct）

### Agent 推理模式有哪些？ReAct 是啥？具体是怎么实现的？

Agent 的推理模式我用过几种。最基础的是直接输出答案，没有中间推理；CoT 是让 LLM 先把推理过程写出来再给答案，准确率更高；ReAct 是在 CoT 基础上加了「行动」，让 LLM 交替输出思考和工具调用，每次行动后再根据结果继续思考，形成一个循环。我觉得 ReAct 是目前 Agent 用得最广的模式，因为它推理过程可见，又能动态利用外部工具，两个优点都有

## **Tools调用**

### 什么是 Function Calling ？原理是什么？

LLM 在没有 Function Calling 之前，想让模型帮你调工具，完全靠解析自然语言。Function Calling 的出现把这件事固定下来了：模型判断需要调工具的时候不输出自然语言，而是直接输出一段结构化的 tool_calls JSON, 你的代码拿到这段 JSON 去真正执行，把结果塞回对话，模型再生成最终答案。

### 什么是 MCP（模型上下文协议）？讲讲它的核心内容？

在 MCP 出现之前，N个Agent要接M个工具，可能就要写 M * N 套接口。MCP 的思路是把这件事标准化：工具提供方按协议实现一个 Server，任何支持 MCP 的 AI 客户端就能直接接进来，一次实现到处复用。协议定义了三类能力：Tools 用于执行有副作用的操作，Resources 是只读数据，Prompts 是提示词模板，底层通信用 JSON-RPC 2.0。我把它理解成给「AI 接工具」这件事定了一套行业标准。

### Skill 是什么？

Agent Skill 是把重复性的工作流程打包成可复用模块的机制。每个 Skill 是一个文件夹，里面有一份 SKILL.md 指令文件，还可以带上脚本、模板、参考文档这些资源。它和普通 prompt 最大的区别是：Skill 能被 Agent 自动发现和按需加载，不用你每次手动输入；

### Function Calling、Skill、MCP 这三个有什么区别？

这三个概念在不同层次工作，不是竞争关系。Function Calling 是最底层的调用协议，解决的是「模型怎么调函数」，模型输出结构化 JSON 告诉程序该调哪个函数、传什么参数。MCP 在 Function Calling 之上做工具标准化，解决的是「工具怎么暴露给模型」，把数据库、API 这些外部能力封装成标准化服务，一次实现到处复用。Agent Skill 在最上层做知识和流程的封装，解决的是「拿到工具之后按什么流程完成任务」，把执行步骤、标准、脚本、模板打包成可复用模块。简单记就是：Function Calling 是语言，MCP 是工具箱，Skill 是操作手册。

## **RAG 知识库**

### 什么是 RAG？详细描述一个完整 RAG 系统的详细工作流程？

RAG解决的问题是，LLM 回答不了训练之外的数据。RAG 的做法是在生成答案之前，在外挂知识库里检索相关内容，然后和用户的问题一起交给 LLM，让它基于这些上下文来回答。

一个完整的 RAG 系统分**离线**和**在线**两个阶段

离线阶段：

**文档加载 ** **文档切割（Chunking）**  **Embedding（向量化）**  **入库**

在线阶段：

**Query 改写**  **向量检索（粗排）**  **Rerank（精排）**  **生成**

### Chunking策略

**初期问题**：用**固定大小分块**（如每512 token），容易在句子中间切断，导致**语义断裂**，检索不准。

**优化方案**：改用**语义分块 (Semantic Chunking)**。

怎么做

1. 先按段落初步分段。
2. 计算相邻段落的 **Embedding 余弦相似度**。
3. 如果相似度高，就**合并**；如果相似度突然下降，就**切分**。

**效果**：保证每个 Chunk 都是一个完整的语义单元，显著提升检索质量。



### **文档更新或删除如何处理**

**难点**：向量数据库不像 MySQL 那样支持简单的 `UPDATE`。

删除方案

- **基于 Metadata 过滤删除**：上传时为每个 Chunk 打上唯一标识（如 `doc_id`）。删除时，根据 `doc_id` 批量删除中所有相关 Chunk。

更新方案

- **先删后增**：先根据 `doc_id` 删除旧的所有 Chunk，再重新解析、切片、向量化，插入新的 Chunk。
- **版本号控制**：在 Metadata 中记录 `version`，查询时只召回最新版本的数据。

###  缓存删除失败怎么办？

**场景**：先更新 DB，再删除 Cache。如果删除 Cache 失败了，会导致脏数据。

解决方案

1. **重试机制**：将删除失败的 Key 放入消息队列，进行异步重试删除。
2. **Binlog 订阅**：使用 Canal 监听 MySQL Binlog，一旦数据变更，自动触发 Cache 删除，解耦业务代码。
3. **TTL 兜底**：给缓存设置过期时间，即使删除失败，数据最终也会过期，保证**最终一致性**。

### 回溯

#### 模板

    class Solution {
        List<List<Integer>> ans = new ArrayList<>();
        List<Integer> path = new ArrayList<>();
    public List<List<Integer>> solve(int[] nums) {
        boolean[] used = new boolean[nums.length]; // 或 int start = 0
        backtrack(nums, used); // 或 backtrack(nums, start)
        return ans;
    }
    
    void backtrack(int[] nums, boolean[] used) {
        // 结束条件：收集结果
        if (path.size() == nums.length) {
            ans.add(new ArrayList<>(path));
            return;
        }
    
        for (int i = 0; i < nums.length; i++) {
            // 剪枝条件：跳过非法选择
            if (used[i]) continue;
    
            // 做选择
            path.add(nums[i]);
            used[i] = true;
    
            // 递归
            backtrack(nums, used);
    
            // 撤销选择（回溯）
            path.remove(path.size() - 1);
            used[i] = false;
        }
    }
}
